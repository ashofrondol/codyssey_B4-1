1 시각화로 보는 전체 구조

코드를 읽기 전에 이 5장의 그림으로 “무엇이 무엇과 연결되는지” 먼저 감을 잡으세요.

① 파일 구조 한눈에 보기

역할별로 폴더가 나뉘어 있습니다. HTML은 1개, CSS·JS는 “기능별”로 잘게 나눠 두었습니다.

2 index.html — 페이지의 뼈대

HTML은 “무엇이 어디에 있는지”를 정하는 골격입니다. 의미가 드러나는 시맨틱 태그로 작성했습니다.

index.html 페이지 구조

브라우저가 가장 먼저 읽는 파일입니다. 화면에 보일 내용의 뼈대를 담고, 머리(<head>)에서 CSS와 JS를 불러옵니다.

① 문서 머리 <head> — 설정과 파일 연결

<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>JeongSeYoung | Portfolio</title>

<!DOCTYPE html>는 “이건 최신 HTML 문서야”라는 선언. lang="ko"는 한국어 페이지라는 뜻(접근성·검색에 도움). charset=UTF-8은 한글이 깨지지 않게, viewport는 모바일에서 화면 크기에 맞게 보이도록 하는 필수 설정입니다.

  <link rel="stylesheet" href="css/style.css" />

  <script src="js/config.js" defer></script>
  <script src="js/theme.js" defer></script>
  ... (기능 모듈들) ...
  <script src="js/main.js" defer></script>

<link> 한 줄로 CSS 진입점 style.css를 연결합니다(나머지 CSS는 그 안에서 합쳐짐). <script ... defer>는 JS를 연결하되, defer 덕분에 HTML을 다 읽은 뒤 적힌 순서대로 실행됩니다. 그래서 공용 도구 config.js가 맨 위, 전체를 깨우는 main.js가 맨 아래입니다.

② 헤더 <header> — 로고·메뉴·버튼

<header class="header" id="header">
  <nav class="nav container" aria-label="주 메뉴">
    <a href="#hero" class="nav__logo">...</a>
    <ul class="nav__menu" id="navMenu">
      <li><a href="#about" class="nav__link">About</a></li>
      ... 메뉴 링크들 ...
    </ul>
    <button class="theme-toggle" id="themeToggle">🌙</button>
    <button class="hamburger" id="hamburger">...</button>
  </nav>
</header>

<header>는 페이지 머리말, <nav>는 내비게이션(메뉴) 묶음입니다. 메뉴 링크의 href="#about"는 “같은 페이지의 id="about" 위치로 이동”을 뜻하는 앵커 링크입니다. 각 요소에 붙은 id(#themeToggle 등)는 나중에 JavaScript가 “이 버튼을 잡아라”라고 찾을 때 쓰는 이름표입니다.

③ 본문 <main> — 섹션들

<main>
  <section class="hero" id="hero"> ... 인사말 + CTA 버튼 ... </section>
  <section id="about">  <article class="about"> ... 자기소개 ... </article> </section>
  <section id="skills"> <ul id="skills-list"></ul> </section>     <!-- JS가 채움 -->
  <section id="projects"> <div id="projects-container"></div> </section> <!-- JS가 채움 -->
  <section id="contact"> <form id="contactForm"> ... </form> </section>
</main>

<main>은 페이지의 핵심 내용(한 페이지에 하나). <section>은 주제별 구획, <article>은 그 자체로 의미가 완결되는 독립 콘텐츠(자기소개·프로젝트 카드)에 씁니다. Skills와 Projects는 내용이 비어 있는데, 이 빈 그릇(#skills-list, #projects-container)을 나중에 JavaScript가 데이터로 채웁니다.

④ 문의 폼 — label과 input의 짝

<div class="form__field">
  <label for="email">이메일</label>
  <input type="email" id="email" name="email" required />
  <small class="form__error" data-error-for="email"></small>
</div>

<label for="email">의 for 값과 <input id="email">의 id가 같으면 둘이 “짝”이 됩니다(라벨을 눌러도 입력칸이 선택됨 — 접근성↑). required는 “필수 입력”, type="email"은 이메일 형식 칸이라는 의미. 빈 <small data-error-for="email">은 에러 메시지가 들어갈 자리이고, data-error-for는 JS가 “이 입력칸의 에러는 여기에 써라”라고 찾는 표시입니다.

🔗 연결: 여기 적힌 id들(#header #themeToggle #navMenu #skills-list #projects-container #contactForm …)을 JavaScript가 $('#아이디')로 찾아 이벤트를 걸거나 내용을 채웁니다. CSS는 class를 보고 꾸밉니다.

3 css/ — 꾸미기와 배치

CSS는 “어떻게 보일지”를 정합니다. 한 덩어리였던 스타일을 기능별 14개 파일로 나눴습니다.

css/style.css진입점

실제 규칙은 없고, 다른 13개 파일을 순서대로 불러오기만 하는 “목차” 파일입니다.

/* 순서가 중요! 위에서부터 적용되고, 아래가 위를 덮어씀(cascade) */
@import "tokens.css";      /* ① 변수 — 가장 먼저 */
@import "base.css";        /* ② 기본 초기화 */
@import "header.css";      /* ⑤ 헤더 ... */
@import "responsive.css";  /* ⑬ 반응형 — 가장 마지막 */

@import는 “다른 CSS 파일을 여기로 끌어와 합쳐라”. 순서가 중요한 이유는 CSS의 cascade(계단식) 규칙 때문입니다. 변수(tokens)는 모두가 참조하므로 맨 앞, 화면 크기별 예외(responsive)는 기본값을 덮어써야 하므로 맨 뒤입니다.

🔗 나머지 13개 파일은 각각 한 가지 역할만 담당합니다: tokens(변수) · base(초기화) · layout(골격) · buttons · header · hero/about/skills/projects/form/footer(섹션별) · widgets(스크롤탑·애니메이션) · responsive(미디어쿼리).

css/tokens.css디자인 변수

색·간격·폰트 같은 값을 변수로 한 곳에 모아 둡니다. 여기만 바꾸면 사이트 전체가 바뀝니다.

:root{                          /* :root = 문서 전체에서 쓸 수 있는 범위 */
  --color-primary: #5b6cff;     /* 변수 정의: --이름: 값 */
  --space-4: 1rem;
}
[data-theme="dark"]{            /* 다크 모드일 때 같은 변수명을 다른 값으로 */
  --color-bg: #0f1117;
}

--color-primary처럼 --로 시작하면 CSS 변수입니다. 다른 곳에서 var(--color-primary)로 꺼내 씁니다. 다크 모드의 핵심: [data-theme="dark"] 안에서 같은 변수명을 어두운 값으로 다시 정의하면, JS가 이 속성을 켜는 순간 변수를 쓰는 모든 요소의 색이 한꺼번에 바뀝니다. (실제 색을 켜는 일은 theme.js가 합니다.)

css/header.css & projects.cssFlexbox vs Grid

초보자가 가장 헷갈리는 두 배치 도구입니다. “한 줄 정렬은 Flex, 격자 배치는 Grid”로 기억하세요.

/* header.css — Flexbox: 한 줄에서 양 끝 정렬 */
.nav{
  display: flex;                      /* 자식들을 가로 한 줄로 */
  justify-content: space-between;     /* 로고는 왼쪽, 메뉴는 오른쪽 끝으로 */
  align-items: center;               /* 세로 가운데 정렬 */
}

Flexbox는 1차원(한 줄) 정렬 도구. justify-content: space-between이 “양 끝으로 밀어내기”라서 로고와 메뉴가 자동으로 좌우로 갈라집니다.

/* projects.css — Grid: 카드들을 격자로 자동 배치 */
.projects{
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(280px, 1fr));
}

Grid는 2차원(행·열) 격자 도구. repeat(auto-fit, minmax(280px, 1fr))는 마법의 한 줄입니다: “각 칸은 최소 280px, 공간이 남으면 똑같이 나눠 늘리고(1fr), 화면이 넓으면 칸 개수를 자동으로 늘려라”. 덕분에 미디어쿼리 없이도 화면 폭에 따라 카드가 1줄→2줄→3줄로 알아서 바뀝니다.

css/responsive.css반응형 (모바일 퍼스트)

기본 스타일은 모바일 기준으로 쓰고, 화면이 커지면 규칙을 더합니다.

/* 기본(모바일): 메뉴 숨김 — header.css에 있음 */
.nav__menu{ display: none; }

/* 768px 이상(태블릿~)에서만 메뉴를 펼치고 햄버거를 숨김 */
@media (min-width: 768px){
  .nav__menu{ display: flex; }
  .hamburger{ display: none; }
}

@media (min-width: 768px)는 “화면 너비가 768px 이상일 때만 이 규칙을 적용”이라는 조건문입니다. 작은 화면을 기본으로 두고 큰 화면에서 살을 붙이는 방식이 모바일 퍼스트. 그래서 좁은 화면에선 메뉴가 숨고 햄버거(☰)가 보입니다.

css/form.css버그 수정 사례 📌

“제출도 안 했는데 성공 메시지가 보이던” 버그를 고친 부분입니다. CSS 우선순위를 이해하기 좋은 예시입니다.

.form__success{ display: flex; }          /* 평소 모양 */

/* 이 한 줄이 핵심 수정: hidden 속성이 display:flex 를 이기게 함 */
.form__success[hidden]{ display: none; }

HTML의 hidden 속성은 원래 요소를 숨기지만(display:none), 우리가 쓴 .form__success{display:flex}가 그걸 덮어써서 항상 보였습니다. 해결: .form__success[hidden]처럼 더 구체적인 선택자로 다시 display:none을 지정하면 이깁니다(구체적일수록 우선). 이제 제출에 성공했을 때만 메시지가 보입니다.

4 js/ — 페이지를 움직이게 하기

JavaScript는 “사용자가 행동하면 무슨 일이 일어날지”를 정합니다. 파일마다 한 가지 기능을 맡습니다. 공통 약속: 모든 모듈은 { init() {...} } 모양의 객체이고, main.js가 init()을 불러 시동을 겁니다.

js/config.js공용 설정·도구

모든 파일이 함께 쓰는 “공구함”입니다. 설정값(CONFIG)과 자주 쓰는 도구 3개를 정의합니다. 가장 먼저 로드됩니다.

핵심 개념

  • 객체(object) — { 키: 값 } 형태로 여러 값을 묶는 상자
  • 화살표 함수 — (인자) => 결과, function을 짧게 쓴 형태
  • querySelector — CSS 선택자로 HTML 요소를 찾아오는 기본 도구

  STORAGE_KEY_THEME: 'portfolio-theme', // 다크모드 저장 이름표
};

const는 “바뀌지 않을 값”을 담는 상자(변수). 여러 설정을 CONFIG라는 객체 하나에 모아두면, 나중에 숫자를 바꾸고 싶을 때 코드 속을 헤맬 필요 없이 여기 한 곳만 고치면 됩니다. (이런 값을 “매직 넘버” 대신 설정으로 빼는 습관)

const $  = (sel, root = document) => root.querySelector(sel);
const $$ = (sel, root = document) => Array.from(root.querySelectorAll(sel));

자주 쓰는 긴 명령을 짧은 별명으로 만든 도구 함수입니다. $('.클래스') = “이 선택자에 맞는 첫 번째 요소 하나 찾기”(querySelector), $$('.클래스') = “맞는 요소 전부를 배열로 찾기”(querySelectorAll). root = document는 “안 주면 문서 전체에서 찾아라”라는 기본값입니다. 이름이 $인 건 그냥 “짧아서” — 특별한 문법이 아닙니다.

function escapeHtml(str){
  return String(str)
    .replace(/&/g, '&amp;').replace(/</g, '&lt;')   // < 같은 특수문자를 안전하게 바꿈
    .replace(/>/g, '&gt;') ... ;
}

GitHub에서 받아온 글자를 화면에 넣기 전에 위험한 특수문자(< > 등)를 무력화하는 안전장치입니다. 이렇게 안 하면 남이 만든 저장소 이름에 악성 코드가 들어 있을 때 실행될 수 있습니다(이것을 XSS 공격이라 함). .replace(정규식, 바꿀값)로 치환합니다.

🔗 여기 정의한 CONFIG · $ · $$ · escapeHtml은 모든 다른 JS 파일이 그대로 가져다 씁니다. (스크립트들이 같은 전역 공간을 공유하기 때문에 가능 — 그래서 config.js를 맨 먼저 로드합니다.)

js/theme.js다크 모드

🌙 버튼을 누르면 밝은/어두운 테마를 전환하고, 선택을 저장해 새로고침 후에도 기억합니다. “이벤트 → 상태 → 렌더” 패턴의 가장 간단한 예시입니다.

핵심 개념

  • state(상태) — 화면을 결정하는 “현재 데이터”. 여기선 { theme: 'light' | 'dark' }
  • localStorage — 브라우저에 값을 저장(새로고침·재방문해도 남음)
  • setAttribute — HTML 요소의 속성을 JS로 바꾸기

코드 따라가기

const Theme = {
  state: { theme: 'light' },              // ① 현재 상태(데이터)

  setState(partial){
    this.state = { ...this.state, ...partial };  // 상태를 새 값으로 합치고
    this.render();                               // 곧바로 화면을 다시 그림
  },

모든 “이벤트→상태→렌더” 모듈의 심장입니다. setState는 “상태를 바꾼 뒤 반드시 render를 부른다”는 약속. { ...this.state, ...partial }의 ...은 스프레드로 “기존 값을 펼쳐 복사하고 바뀐 부분만 덮어쓰기”를 뜻합니다. this는 “이 객체 자신(Theme)”을 가리킵니다.

  init(){
    const saved = localStorage.getItem(CONFIG.STORAGE_KEY_THEME); // 저장된 값 읽기
    const systemDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
    this.setState({ theme: saved || (systemDark ? 'dark' : 'light') });
    $('#themeToggle').addEventListener('click', () => this.toggle()); // 버튼에 클릭 연결
  },

init()은 시작할 때 한 번 실행됩니다. 저장된 테마가 있으면 그걸, 없으면(|| = “왼쪽이 없으면 오른쪽”) 시스템 설정을 따릅니다. addEventListener('click', 함수)는 “이 버튼이 클릭되면 이 함수를 실행해줘”라고 등록하는 것 — HTML에 onclick을 쓰지 않는 이유가 바로 이 깔끔한 분리 때문입니다.

  toggle(){
    const theme = this.state.theme === 'dark' ? 'light' : 'dark'; // 반대로 뒤집기
    this.setState({ theme });                                     // ② 상태 변경 → render 자동
    localStorage.setItem(CONFIG.STORAGE_KEY_THEME, theme);        // 선택 저장
  },
  render(){
    document.documentElement.setAttribute('data-theme', this.state.theme); // ③ <html>에 표시
  },

toggle()은 A === 'dark' ? 'light' : 'dark'라는 삼항 연산자(조건 ? 참값 : 거짓값)로 테마를 뒤집습니다. render()는 <html>에 data-theme="dark"를 붙입니다 → 그 순간 tokens.css의 다크 변수가 켜지며 화면 전체 색이 바뀝니다. JS는 “스위치만 켜고”, 실제 색은 CSS가 담당하는 멋진 분업입니다.

🔗 CONFIG(config.js)에서 저장 키를 가져오고, 색 변경은 tokens.css의 [data-theme="dark"]가 처리합니다.

js/menu.js햄버거 메뉴

모바일에서 ☰ 버튼을 누르면 메뉴가 펼쳐지고, 다시 누르면 닫힙니다.

핵심 개념

  • classList.toggle — 클래스를 “있으면 빼고, 없으면 넣는” 스위치
  • aria-* 속성 — 스크린리더 같은 보조기기를 위한 상태 안내

코드 따라가기

btn.addEventListener('click', () => {
  const isOpen = menu.classList.toggle('is-open');   // 클래스 켜고/끄기 + 결과(true/false) 받기
  btn.classList.toggle('is-open', isOpen);           // 버튼 모양도 같이 (☰ ↔ ✕)
  btn.setAttribute('aria-expanded', String(isOpen)); // 보조기기에 열림 상태 알림
});

classList.toggle('is-open')가 핵심입니다. is-open 클래스를 붙였다 뗐다 하면, CSS(header.css)에 미리 적어둔 .nav__menu.is-open{ display:flex } 규칙이 켜졌다 꺼졌다 하며 메뉴가 열리고 닫힙니다. toggle은 결과를 true/false로 돌려주므로 그 값(isOpen)으로 버튼 모양과 안내까지 한 번에 맞춥니다.

🔗 JS는 is-open 클래스만 붙일 뿐, “펼쳐 보이는” 실제 모양은 header.css가 그립니다.

js/scroll.js스크롤 반응

스크롤에 따라 ① 네비 배경을 바꾸고 ② 맨 위로 가기 버튼을 보였다 숨기고 ③ 메뉴 클릭 시 부드럽게 이동시킵니다.

핵심 개념

  • scroll 이벤트 — 스크롤할 때마다 발생(아주 자주!)
  • requestAnimationFrame — 너무 잦은 실행을 “화면 갱신 주기”에 맞춰 솎아내는 최적화
  • scrollIntoView — 특정 요소가 보이도록 스르륵 이동

코드 따라가기

let ticking = false;
window.addEventListener('scroll', () => {
  if (ticking) return;                       // 이미 예약돼 있으면 건너뜀(과부하 방지)
  window.requestAnimationFrame(() => {
    const y = window.scrollY;                // 현재 스크롤 위치(px)
    header.classList.toggle('is-scrolled', y > CONFIG.NAV_SCROLL_THRESHOLD); // 60px↑
    scrollTopBtn.classList.toggle('is-visible', y > CONFIG.SCROLL_TOP_THRESHOLD); // 300px↑
    ticking = false;
  });
  ticking = true;
});

scroll 이벤트는 1초에 수십~수백 번 일어나서, 매번 일을 하면 버벅입니다. requestAnimationFrame은 “다음 화면 그릴 때 딱 한 번만 실행”하도록 묶어주는 최적화 기법(쓰로틀링)입니다. classList.toggle('클래스', 조건)의 두 번째 인자는 “조건이 참이면 붙이고 거짓이면 떼라”는 뜻 — y > 60이면 네비에 배경이 생깁니다.

$$('a[href^="#"]').forEach((link) => {
  link.addEventListener('click', (event) => {
    event.preventDefault();                  // 순간이동(기본 동작) 막고
    $(link.getAttribute('href')).scrollIntoView({ behavior: 'smooth' }); // 부드럽게 이동
  });
});

$$('a[href^="#"]')는 “href가 #으로 시작하는 모든 링크”를 찾습니다(^= = “~로 시작”). forEach로 링크 하나하나에 클릭 이벤트를 겁니다. event.preventDefault()는 링크의 기본 동작(딱 점프)을 막고, 대신 scrollIntoView({behavior:'smooth'})로 스르륵 이동시킵니다.

🔗 기준 숫자(60·300)는 CONFIG(config.js)에서 가져오고, is-scrolled · is-visible 클래스의 실제 모양은 CSS가 그립니다.

js/typing.js타이핑 효과

Hero(첫 화면)의 이름이 한 글자씩 타자기처럼 나타나게 합니다.

핵심 개념

  • setTimeout — “잠시 뒤에 이 함수를 실행해줘” (지연 실행)
  • 문자열 .slice(0, n) — 글자를 0번째부터 n개까지 잘라내기

코드 따라가기

const fullText = target.textContent.trim();  // 원래 글자 전체 저장
target.textContent = '';                     // 화면은 일단 비움
let i = 0;
const tick = () => {
  if (i <= fullText.length){
    target.textContent = fullText.slice(0, i); // 0~i 글자만 보여주고
    i += 1;
    setTimeout(tick, 90);                       // 90ms 뒤 자기 자신을 또 호출
  }
};
tick();

함수가 자기 자신을 다시 부르는(재귀) 방식입니다. i를 1씩 늘리며 slice(0, i)로 “0번째부터 i개”만 보여주면, 90ms마다 글자가 하나씩 늘어 타이핑처럼 보입니다. let은 const와 달리 값이 바뀌는 변수에 씁니다(i가 계속 증가하니까).

js/skills.js기술 목록 그리기

기술 스택 데이터(배열)를 HTML 카드로 변환해 빈 <ul>에 채웁니다. 데이터→화면 변환의 가장 단순한 예시라 꼭 이해하면 좋습니다.

핵심 개념

  • 배열(array) — [ ... ]로 여러 값을 순서대로 담은 목록
  • map() — 배열의 각 항목을 “다른 모양으로 변환”해 새 배열로
  • 템플릿 리터럴 — 백틱(`)으로 문자열 안에 ${값}을 끼워 넣기

코드 따라가기

data: [
  { name: 'HTML5', icon: 'fa-brands fa-html5' },   // 객체들이 담긴 배열
  { name: 'CSS3',  icon: 'fa-brands fa-css3-alt' },
  ...
],
init(){
  list.innerHTML = this.data
    .map(({ name, icon }) => `                     // 항목마다 HTML 한 조각 생성
      <li class="skill reveal">
        <div class="skill__icon"><i class="${icon}"></i></div>
        <div class="skill__name">${name}</div>
      </li>`)
    .join('');                                     // 조각들을 하나로 합침
}

3단 콤보입니다. ① .map(...)이 배열의 각 객체를 HTML 문자열로 바꾸고 ② ({ name, icon })는 구조분해로 객체에서 필요한 값만 꺼내며 ③ 백틱 `...${name}...`(템플릿 리터럴)으로 값을 문자열에 끼워 넣습니다. map의 결과는 “문자열들의 배열”이라 .join('')으로 한 덩어리로 붙인 뒤 innerHTML에 넣어 화면에 그립니다.

🔗 index.html의 빈 <ul id="skills-list">를 채웁니다. 만든 li에 reveal 클래스를 붙여 reveal.js의 등장 애니메이션 대상이 되게 합니다.

js/reveal.js스크롤 등장 애니메이션

요소가 화면에 들어오는 순간 스르륵 나타나게 합니다. 스크롤 위치를 직접 계산하지 않고 브라우저에게 감시를 맡기는 똑똑한 방법을 씁니다.

핵심 개념

  • IntersectionObserver — “이 요소가 화면에 보이기 시작하면 알려줘”라고 브라우저에 부탁하는 감시자
  • 콜백(callback) — “나중에 어떤 일이 생기면 실행할 함수”

코드 따라가기

this.observer = new IntersectionObserver((entries) => {
  entries.forEach((entry) => {
    if (entry.isIntersecting){                  // 화면에 들어왔다면
      entry.target.classList.add('is-visible'); // 보이게 하는 클래스 추가
      this.observer.unobserve(entry.target);    // 한 번 보였으면 감시 해제
    }
  });
}, { threshold: CONFIG.REVEAL_THRESHOLD });     // 20% 보이면 발동

IntersectionObserver는 “요소가 화면에 들어왔는지”를 브라우저가 알아서 감시하다가, 들어오면 우리가 준 함수(콜백)를 실행합니다. 스크롤 좌표를 일일이 계산하는 것보다 훨씬 효율적입니다. 들어오면 is-visible 클래스를 붙여(CSS가 투명→불투명 + 위로 떠오름) 애니메이션이 시작되고, unobserve로 그 요소는 더 이상 감시하지 않습니다(한 번만 나타나면 되니까). threshold 0.2 = “20% 정도 보이면 발동”.

🔗 class="reveal"이 붙은 모든 요소가 대상입니다. projects.js가 카드를 새로 그린 뒤 Reveal.observe()를 다시 불러 새 카드도 감시 목록에 넣습니다.

js/navspy.js현재 위치 메뉴 강조

스크롤해서 보고 있는 섹션에 해당하는 메뉴 링크를 자동으로 강조(밑줄)합니다.

핵심 개념

  • IntersectionObserver + rootMargin — 감시 영역을 “화면 중앙 띠”로 좁히기
  • classList.toggle(클래스, 조건) — 조건에 따라 강조를 켜고 끄기

코드 따라가기

const observer = new IntersectionObserver((entries) => {
  entries.forEach((entry) => {
    if (entry.isIntersecting){
      const id = entry.target.id;                // 지금 보이는 섹션의 id
      links.forEach((link) =>
        link.classList.toggle('is-active',
          link.getAttribute('href') === `#${id}`)); // 그 섹션 가리키는 링크만 강조
    }
  });
}, { rootMargin: '-40% 0px -55% 0px' });          // 화면 중앙 띠에서만 감지

reveal.js와 같은 도구지만 목적이 다릅니다. 어떤 섹션이 화면 중앙에 오면, 모든 메뉴 링크를 돌며 그 섹션을 가리키는 링크에만 is-active를 켭니다(===로 href 비교, 나머지는 끔). rootMargin으로 감지 영역을 위아래로 줄여 “화면 한가운데 들어온 섹션”만 활성화되게 했습니다.

js/projects.jsGitHub API 연동 ⭐

이 프로젝트에서 가장 중요한 파일입니다. 인터넷에서 GitHub 저장소 목록을 가져와 카드로 그리고, 그 사이의 로딩·성공·에러·빈 상태를 모두 처리합니다. (위 ④번 “상태 머신” 그림을 코드로 옮긴 것)

핵심 개념

  • 비동기(async/await) — 인터넷 요청처럼 “시간이 걸리는 일”을 기다리는 방법
  • fetch — 인터넷 주소에서 데이터를 가져오는 명령
  • try/catch — 실패할 수 있는 일을 감싸 “성공/실패”로 갈래 나누기
  • filter / map / 구조분해 — 데이터를 거르고, 카드로 바꾸기

코드 따라가기

state: { status: 'idle', repos: [], activeLang: 'All', error: null },

setState(partial){
  this.state = { ...this.state, ...partial };  // 상태 합치고
  this.render();                               // 상태에 맞는 화면 그림
},

theme.js와 똑같은 패턴이지만 상태가 더 풍부합니다. status는 지금 어느 단계인지(idle/loading/success/error/empty), repos는 받아온 저장소들, activeLang은 선택한 언어 필터, error는 에러 메시지. 이 4개 값이 화면을 결정합니다.

async fetchRepos(){
  this.setState({ status: 'loading' });        // ① 일단 로딩 화면(스피너)
  try{
    const res = await fetch(url);              // ② 인터넷에서 데이터 받기(기다림)
    if(!res.ok){                               // 응답이 실패면(403/404 등) 에러 던지기
      if(res.status === 403) throw new Error('호출 한도 초과...');
      throw new Error(`요청 실패 (HTTP ${res.status})`);
    }
    const data = await res.json();             // ③ 받은 데이터를 JS가 쓸 형태로
    const repos = data.filter(r => !r.fork).slice(0, 12); // 남의 복제본 빼고 12개
    if(repos.length === 0) return this.setState({ status: 'empty' });
    this.setState({ status: 'success', repos });    // ④ 성공!
  }catch(err){
    this.setState({ status: 'error', error: err.message }); // ⑤ 어떤 실패든 여기로
  }
}

async가 붙은 함수 안에서는 await를 쓸 수 있습니다 — “이 줄이 끝날 때까지 기다렸다 다음 줄로”. 인터넷 요청은 시간이 걸리니 await fetch(...)로 기다립니다. try{ } 안의 코드 중 하나라도 실패하거나 throw로 에러를 던지면, 즉시 catch(err){ }로 점프합니다. 그래서 네트워크 끊김·403(한도초과)·404(없는 계정) 같은 모든 실패가 한 곳에서 “error 상태”로 모입니다. .filter(r => !r.fork)는 “fork(복제본)가 아닌 것만 남겨라”.

render(){
  const { status, repos, error } = this.state;          // 구조분해로 상태 꺼냄
  if(status === 'loading'){ ...스피너 HTML...; return; }  // 상태마다 다른 화면
  if(status === 'error'){ ...에러+재시도 버튼...; return; }
  if(status === 'empty'){ ...없습니다 메시지...; return; }
  if(status === 'success'){ this.renderFilters(); this.renderCards(); }
}

render()는 status 값만 보고 어떤 화면을 그릴지 결정합니다. 마치 “신호등”처럼, 상태가 바뀌면(setState) 자동으로 이 함수가 불려 알맞은 화면(스피너/카드/에러/빈)으로 갈아끼웁니다. 이것이 “상태가 바뀌면 화면이 따라온다”는 선언적 UI의 핵심입니다. 에러 화면의 [다시 시도] 버튼에는 fetchRepos를 다시 연결해 재시도가 됩니다.

이름·이메일·메시지를 검사해서, 비었거나 형식이 틀리면 에러를 보여주고, 통과하면 성공 메시지를 띄웁니다. 역시 state 객체 + setState + render 패턴입니다.

핵심 개념

  • submit / blur / input 이벤트 — 제출 / 칸을 벗어남 / 글자 입력
  • event.preventDefault() — 폼의 기본 “새로고침 제출”을 막기
  • 정규식(RegExp) — 글자 패턴(이메일 형식 등)을 검사하는 도구

validate(field){
  const value = field.value.trim();
  if(field.required && value === '') return '필수 입력 항목입니다.';
  if(field.type === 'email' && !this.isValidEmail(value)) return '올바른 이메일 형식이 아닙니다.';

event.preventDefault()가 없으면 폼은 제출 시 페이지를 새로고침해버려 JS 검증이 소용없어집니다 — 그래서 꼭 막습니다. validate는 한 칸을 검사해 “에러 메시지(문제 있을 때) 또는 빈 문자열(정상)”을 돌려주는 순수 함수입니다. &&는 “그리고”(둘 다 참일 때).

render(){
  Object.keys(errors).forEach((id) => {        // 칸마다
    const errorEl = $(`[data-error-for="${id}"]`);
    errorEl.closest('.form__field').classList.toggle('has-error', errors[id] !== '');
    errorEl.textContent = errors[id];          // 에러 글자 표시
  });

render는 state를 화면에 반영만 합니다. 각 칸의 에러 메시지를 data-error-for로 찾은 자리에 쓰고, 문제가 있으면 has-error 클래스를 켜(CSS가 테두리를 빨갛게) 줍니다. 성공 메시지는 status === 'success'일 때만 hidden=false로 보입니다 — 이 한 줄이 “제출도 안 했는데 떠 있던” 문제를 막아 줍니다(앞 CSS 수정과 한 쌍).

참고: 이 폼은 학습용이라 실제 메일을 보내지는 않습니다(검증까지만, 콘솔에 데이터만 출력). 실제 전송을 원하면 Formspree·EmailJS 같은 서비스를 submit()에 연결하면 됩니다.

5 용어 사전

본문에 나온 용어를 한 줄로 정리했습니다. 모르는 단어가 나오면 여기서 찾으세요.

  • DOM — 브라우저가 HTML을 다룰 수 있게 만든 “요소들의 나무 구조”. JS는 이 DOM을 찾고 바꿉니다.
  • 요소(element) — <button>, <div> 같은 HTML 태그 하나하나.
  • querySelector / $ — CSS 선택자(#id, .class)로 요소를 찾는 명령. 이 프로젝트는 $로 줄여 씀.
  • 이벤트(event) — 클릭·입력·스크롤처럼 “사용자나 브라우저가 일으키는 일”.
  • addEventListener — “이 이벤트가 일어나면 이 함수를 실행해줘”라고 등록하는 것.
  • event.preventDefault() — 그 이벤트의 “기본 동작”(링크 점프, 폼 새로고침 등)을 막기.
  • 상태(state) — 화면을 결정하는 “현재 데이터”. 이게 바뀌면 화면을 다시 그립니다.
  • render() — 상태를 보고 화면(DOM)을 그리는 함수.
  • setState() — 상태를 바꾸고 곧바로 render를 부르는 함수(둘을 짝지어 실수 방지).
  • const / let — 값을 담는 상자. const는 안 바뀌는 값, let은 바뀌는 값. (var는 안 씀)
  • 함수(function) — 일을 묶어 이름 붙인 “명령 세트”. 화살표 함수 () => {}는 그 짧은 표기.
  • 객체 / 배열 — 객체 {키:값}는 이름표 달린 묶음, 배열 [ ]는 순서 있는 목록.
  • 구조분해 — const { name } = obj처럼 묶음에서 필요한 값만 꺼내는 문법.
  • 스프레드(...) — {...a, ...b}처럼 펼쳐서 합치거나 복사하는 문법.
  • map / filter / forEach — 배열 도구. map=각 항목 변환, filter=조건으로 거르기, forEach=하나씩 순회.
  • 템플릿 리터럴 — 백틱 `...` 문자열. 안에 ${값}을 끼워 넣을 수 있음.
  • innerHTML / textContent — 요소 안의 내용 바꾸기. innerHTML=HTML째, textContent=순수 글자만(더 안전).
  • classList.add/remove/toggle — 요소의 CSS 클래스를 켜고/끄고/뒤집기. JS와 CSS를 잇는 다리.
  • async / await — 시간이 걸리는 일(인터넷 요청 등)을 “기다렸다 다음 줄”로 자연스럽게 쓰는 문법.
  • fetch — 인터넷 주소에서 데이터를 받아오는 명령. 결과는 기다려야(await) 함.
  • try / catch — 실패할 수 있는 코드를 try로 감싸고, 실패하면 catch로 넘어가 처리.
  • localStorage — 브라우저에 값을 저장. 새로고침·재방문해도 남아 있음(다크모드 기억에 사용).
  • IntersectionObserver — “요소가 화면에 보이면 알려줘”라고 브라우저에 맡기는 감시자.
  • 정규식(RegExp) — /패턴/ 형태로 글자 패턴을 검사(이메일 형식 확인 등).
  • defer — <script> 속성. “HTML 다 읽은 뒤 순서대로 실행”하게 함.
  • CSS 변수 / cascade — --이름으로 값을 모아두는 것 / 나중 규칙이 앞 규칙을 덮는 CSS 적용 순서.
  • Flexbox / Grid — 배치 도구. Flex=한 줄 정렬, Grid=행·열 격자.
  • 미디어쿼리(@media) — “화면이 OO px 이상일 때만”처럼 조건부 CSS.
