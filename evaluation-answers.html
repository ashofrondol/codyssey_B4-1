구조와 문법을 설명할 수 있는가

HTML/CSS/JS 분리, 시맨틱 태그, CSS 변수, addEventListener — “왜 그렇게 했는지”를 말로 설명하는 항목입니다.

Q1 HTML·CSS·JavaScript를 각각 파일로 분리한 이유와 각 파일의 역할은?

관심사의 분리(Separation of Concerns) 때문입니다. 세 언어는 역할이 다릅니다.

  • HTML — 구조/의미: 무엇이 제목이고 무엇이 목록인지(index.html)
  • CSS — 표현/스타일: 어떻게 보이는지(색·간격·레이아웃)
  • JavaScript — 동작/상호작용: 클릭하면 무슨 일이 일어나는지

분리하면 ① 유지보수가 쉽고(스타일만 고칠 때 HTML을 안 건드림), ② 협업이 편하며, ③ 브라우저가 CSS/JS를 캐싱해 재방문 성능이 좋고, ④ 코드가 읽기 쉬워집니다.

저는 여기서 한 발 더 나아가 CSS·JS를 기능별 파일로 한 번 더 분리했습니다. CSS는 tokens / base / layout / header / hero / … 처럼 역할별로 쪼개 style.css에서 @import 하고, JS는 theme.js · menu.js · projects.js … 처럼 기능 단위로 나눠 main.js가 초기화만 담당하게 했습니다. 파일 하나가 한 가지 일만 하도록요.

Q2 시맨틱 태그를 어떤 기준으로 선택했는가?

“이 영역이 의미상 무엇인가”를 기준으로 골랐습니다. div는 의미가 없으므로, 의미가 있는 영역엔 의미 있는 태그를 썼습니다.

  • <header> — 페이지 머리말(로고+네비). 섹션 제목 묶음에도 .section__head로 사용
  • <nav> — 주요 내비게이션 링크 모음
  • <main> — 페이지의 핵심 콘텐츠(문서당 1개)
  • <section> — 주제로 묶이고 제목을 갖는 구획(Hero/About/Skills/Projects/Contact)
  • <article> — 그 자체로 의미가 완결되는 독립 콘텐츠. 자기소개 블록과 GitHub 프로젝트 카드에 사용
  • <footer> — 저작권·소셜 링크 등 바닥글

효과: 접근성(스크린리더가 구조를 읽음), SEO(검색엔진이 구조 이해), 가독성(코드만 봐도 구조가 보임).

Q3 CSS 변수(:root)로 무엇을 정의했고, 변수로 관리하면 어떤 이점이 있는가?

css/tokens.css의 :root에 색상·간격·폰트·반경·모션을 변수로 정의했습니다.

:root{
  --color-primary:#5b6cff;
  --color-text:#1a1d23;
  --space-4:1rem;  --space-6:2rem;
  --font-sans:"Inter","Noto Sans KR",sans-serif;
  --radius-md:12px;  --t-base:250ms;
}

이점:

  • 단일 출처(Single Source of Truth) — 브랜드 색을 바꾸려면 변수 한 줄만 수정하면 전체에 반영됩니다.
  • 다크 모드가 거의 공짜 — [data-theme="dark"]에서 같은 변수명만 다른 값으로 재정의하면 됩니다. 실제 화면 규칙은 그대로 두고 변수 값만 바뀝니다.
  • 일관성 — 간격/색이 제멋대로인 “매직 넘버”가 사라지고 디자인 토큰으로 통일됩니다.
/* 같은 변수명을 다크 값으로만 덮어쓰면 전체 테마가 전환됨 */
[data-theme="dark"]{
  --color-bg:#0f1117;
  --color-text:#e7eaf2;
}

Q4 인라인 onclick 대신 addEventListener를 쓴 이유를 두 방식의 차이로 설명하면?

비교HTML onclick="..."addEventListener
관심사 분리HTML에 JS가 섞임동작은 JS 파일에만 — 구조/동작 분리
핸들러 개수하나만 가능(덮어씀)같은 이벤트에 여러 개 등록 가능
옵션없음{ once, capture, passive } 등 제어 가능
제거어려움removeEventListener로 깔끔히 제거
동적 요소문자열로 박아야 함JS로 생성한 요소에도 동일하게 연결

그래서 이 프로젝트는 HTML에 on* 속성을 하나도 쓰지 않고, 모든 이벤트를 JS에서 addEventListener로 연결했습니다. 동적으로 생성되는 “다시 시도” 버튼·언어 필터 버튼도 렌더 직후 리스너를 붙입니다.

// js/theme.js — 인라인 핸들러 없이 JS에서 연결
$('#themeToggle').addEventListener('click', () => this.toggle());

동작 원리와 흐름을 따라 설명할 수 있는가

Q1 “이벤트 → 상태 변경 → 화면 업데이트” 흐름을 예시로 짚어줄 수 있는가? (다크 모드 예시)

  1. 이벤트 — 사용자가 🌙 버튼 클릭 → Theme.toggle() 호출 (js/theme.js)
  2. 상태 변경 — setState({ theme })로 state.theme을 'light' ↔ 'dark'로 바꾸면 render()가 자동 호출됨 (+ localStorage 저장)
  3. 화면 업데이트 — render()가 <html>에 data-theme="dark"를 설정
  4. 스타일 반영 — css/tokens.css의 [data-theme="dark"]가 변수를 재정의 → 변수를 쓰는 모든 요소 색이 한 번에 바뀜
state = { theme: 'light' };

setState(partial){
  this.state = { ...this.state, ...partial };  // ② 상태 변경
  this.render();                               // ③ 변경되면 무조건 render
}
toggle(){
  const theme = this.state.theme === 'dark' ? 'light' : 'dark';
  this.setState({ theme });                    // 상태 → 렌더 (자동)
  localStorage.setItem(CONFIG.STORAGE_KEY_THEME, theme); // 새로고침 유지
}
render(){
  document.documentElement.setAttribute('data-theme', this.state.theme); // ③ DOM 반영
}

핵심은 “상태를 바꾼 뒤 반드시 render()를 호출”하는 단방향 흐름입니다. Theme·Projects·ContactForm 세 모듈 모두 객체형 state + setState + render 패턴으로 통일했습니다. 이게 바로 React의 상태→렌더링 모델의 기초입니다.

Q2 async/await + try/catch로 API 성공/실패를 어떻게 분기했는가?

async fetchRepos(){
  this.setState({ status:'loading' });          // 요청 시작 → 로딩 UI
  try{
    const res = await fetch(url, {...});          // 응답을 기다림
    if(!res.ok){                                  // HTTP 에러를 직접 throw
      if(res.status === 403) throw new Error('호출 한도 초과...');
      if(res.status === 404) throw new Error('사용자를 찾을 수 없습니다...');
      throw new Error(`요청 실패 (HTTP ${res.status})`);
    }
    const data = await res.json();
    const repos = data.filter(r => !r.fork).slice(0,12);
    if(repos.length === 0) return this.setState({ status:'empty' });
    this.setState({ status:'success', repos });   // 성공 분기
  }catch(err){
    this.setState({ status:'error', error:err.message }); // 모든 실패가 여기로
  }
}

핵심 포인트: await로 비동기를 동기처럼 위에서 아래로 읽히게 하고, 네트워크 오류·HTTP 오류(403/404)·JSON 파싱 오류를 모두 catch 한 곳으로 모아 에러 상태로 전환합니다. !res.ok일 때 throw 해서 성공/실패 흐름을 일원화한 것이 포인트입니다.

📂 js/projects.js — fetchRepos()

Q3 map·filter로 GitHub 데이터를 카드 UI로 바꾸는 과정을 단계별로?

  1. fetch + res.json()으로 저장소 배열을 받음
  2. .filter(r => !r.fork).slice(0,12) — fork 저장소 제외 + 12개로 제한
  3. (보너스 필터) 선택 언어가 있으면 .filter(r => r.language === activeLang)
  4. .map(({name, description, html_url, ...}) => \`...\`) — 구조분해로 필요한 필드만 꺼내 템플릿 리터럴 카드 문자열로 변환
  5. .join('')으로 한 문자열로 합쳐 container.innerHTML에 주입
container.innerHTML = filtered
  .map(({ name, description, html_url, stargazers_count, language }) => `
    <article class="card reveal">
      <h3 class="card__title">${escapeHtml(name)}</h3>
      <p>${escapeHtml(description || '설명 없음')}</p>
      <a href="${html_url}" target="_blank">저장소 보기</a>
    </article>`)
  .join('');
외부에서 받은 값(name·description)은 escapeHtml()로 이스케이프해 innerHTML XSS를 막았습니다.

Q4 Flexbox와 Grid를 각각 어디에 적용했고, 왜 그 방식을 골랐는가?

방식적용 위치선택 이유
Flexbox
(1차원)		 .nav(로고↔메뉴 양끝), .nav__actions, .hero__cta, .card 내부 세로 정렬 한 줄(가로 또는 세로) 안에서 요소를 정렬·분배할 때. 예: 로고는 왼쪽, 메뉴는 오른쪽(justify-content:space-between)

“왜 이렇게 설계했는가”를 묻는 항목입니다.

Q1 상태(STATE)를 객체로 따로 만들어 관리한 이유는? 그냥 변수로 하면 안 되는가?

관련된 상태를 하나의 객체로 묶고, 그 객체를 바꿀 때 항상 render()가 따라오게 하기 위해서입니다. js/projects.js가 대표 예시입니다.

state = { status:'idle', repos:[], activeLang:'All', error:null };

setState(partial){
  this.state = { ...this.state, ...partial };  // 상태를 한곳에서 갱신
  this.render();                               // 갱신되면 '무조건' 다시 그림
}

객체+setState의 장점:

  • 단일 출처: 화면을 결정하는 값들이 한 객체(status·repos·activeLang)에 모여 있어 추적이 쉬움
  • 갱신 누락 방지: 상태 변경 통로가 setState 하나라, “바꿨는데 화면을 안 그림” 버그가 구조적으로 막힘
  • React로의 연결: 이 패턴이 곧 useState/setState → 리렌더의 축소판

그냥 변수로 흩어두면 let status, let repos … 가 여기저기 흩어지고, 값을 바꿀 때마다 화면 갱신 코드를 직접 호출해야 해서 빠뜨리기 쉽고, 어디서 상태가 바뀌는지 추적이 어려워집니다.

✅ 세 모듈 모두 동일 패턴으로 통일했습니다.
  • Theme.state = { theme } — 토글 시 setState({ theme }) → render()가 html[data-theme] 반영
  • Projects.state = { status, repos, activeLang, error } — fetch/필터/재시도 → setState → 상태별 UI 렌더
  • ContactForm.state = { errors, status } — blur/input/submit → setState → 필드 에러·성공 메시지 렌더 (DOM을 직접 상태원으로 쓰지 않음)
따라서 “왜 STATE 객체로 관리하나”라는 질문에, 세 기능 모두 같은 근거(단일 출처·갱신 누락 방지)로 일관되게 설명할 수 있습니다.

Q2 반응형 디자인을 “모바일 퍼스트”로 작성한 이유는?

기본 스타일을 모바일 기준으로 먼저 작성하고, min-width 미디어쿼리로 화면이 커질 때 스타일을 더하는 방식입니다.

/* 기본 = 모바일 (네비 숨김) */
.nav__menu{ display:none; }
/* 768px 이상에서 '추가' */
@media (min-width:768px){ .nav__menu{ display:flex; } .hamburger{ display:none; } }

이유:

  • 점진적 향상: 가장 제약이 큰 환경(작은 화면)을 먼저 만족시키고, 여유가 생기는 큰 화면에서 살을 붙임 → CSS가 “덮어쓰기”보다 “쌓기”가 되어 단순해짐
  • 사용자 비중: 실제 트래픽에서 모바일 비중이 높음
  • 핵심 우선: 좁은 화면에서 꼭 필요한 콘텐츠/구조를 먼저 정리하게 됨
  • 성능: 모바일 기기가 불필요한 데스크톱 전용 규칙을 덜 처리

📂 기본(모바일) 스타일은 각 파일 · 확장은 css/responsive.css(min-width:768px / 1024px)
